Claude Code がこのプロジェクトで作業する際に守るべきルールと設定をまとめたファイルです。

- 不明点があれば実装前に必ず確認する

## プロジェクト概要

昆虫食初心者向けのFlutterアプリ。診断機能・昆虫一覧・昆虫詳細の3機能をMVPとして開発中。

| 項目 | 内容 |
|------|------|
| フロントエンド | Flutter + Dart |
| バックエンド | Go + Gin（Cloud Run） |
| DB | TiDB Cloud（MySQL互換） |
| AI | Claude Haiku 4.5 |

設計書は `docs/` 配下に配置している。作業前に必要なドキュメントを参照すること。

| ドキュメント | ファイル |
|------------|--------|
| 要件定義 | `../docs/01_requirements.md` |
| アーキテクチャ | `../docs/02_architecture.md` |
| DB設計 | `../docs/03_database.md` |
| API設計 | `../docs/04_api.md` |
| サイトマップ | `../docs/05_sitemap.md` |

## コミットメッセージ

- **1行・日本語・シンプルに**書く
- フォーマット：`<動詞>: <変更内容の要約>`

| 動詞 | 用途 |
|------|------|
| `追加` | 新機能・新ファイルの追加 |
| `修正` | バグ修正 |
| `更新` | 既存機能の改善・変更 |
| `削除` | ファイル・コードの削除 |
| `リファクタリング` | 動作を変えない構造改善 |
| `ドキュメント` | README・設計書などの更新 |
| `テスト` | テストコードの追加・修正 |

**例：**
```
追加: 診断フロー画面を実装
修正: 昆虫詳細画面のレーダーチャートが表示されない問題を修正
ドキュメント: API設計書にエラーレスポンス仕様を追記
```

## コード・ドキュメント生成スタイル

### Dart / Flutter

- `flutter analyze` で警告が出ないコードを書く（`analysis_options.yaml` の `flutter_lints` に準拠）
- 状態管理は **Riverpod** を使用する（`../docs/02_architecture.md` 参照）
- フォルダ構成は `../docs/02_architecture.md` の「Flutter構成」に従う（`features/<機能>/{models, repositories, services, pages}`）
- HTTP通信は `dio` を使用し、`shared/api/api_client.dart` に共通処理をまとめる
- 画面遷移は `go_router` を使用する
- ウィジェットは `shared/widgets/` に共通部品として切り出す
- 1つのウィジェット・クラスが肥大化しないよう、適切に分割する

### コメント

- **自明なコードにコメントは不要**。ロジックが複雑な箇所のみ書く
- コメントは日本語で記述する
- `TODO:` コメントは残さず、その場で実装する

## Claude に意識してほしい観点

### セキュリティ

- APIキーや認証情報をコードに直接書かない（`.env` や環境変数を使用する）
- 機密情報を含むファイルは必ず `.gitignore` に追加する
- ユーザー入力値はバリデーションしてから使用する（スコアの範囲チェックなど）
- XSS・インジェクション系の脆弱性を作り込まない

### コード品質（やらないこと）

- 現時点で不要な将来機能の先取り実装
- 1箇所でしか使わないウィジェット・関数の抽象化
- 既存の動作に影響しない不要なリファクタリング
- 求められていないエラーハンドリングの追加

### エラーハンドリング

- Claude API呼び出しは**リトライ3回・失敗時はデフォルトコメントを表示**する（`../docs/01_requirements.md` 参照）
- API通信のエラーはユーザーに分かりやすいメッセージで表示する
- エラーの握りつぶし（空の `catch` ブロック）はしない

### API連携

- バックエンドのAPIエンドポイントは `../docs/04_api.md` に定義されている仕様に従う
- リクエスト/レスポンスの型は `../docs/04_api.md` のJSONスキーマを参照して定義する
